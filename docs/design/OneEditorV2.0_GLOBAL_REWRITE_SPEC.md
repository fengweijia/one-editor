# OneEditor v2.0 — 产品设计文档

---

## 一、核心价值主张

**把"读"和"拆"文章的时间压缩 80%，构建一个可检索的个性化灵感写作系统。**

本质不是一个"AI 摘要工具"，而是一个 **"阅读代理"** —— 它帮你读、帮你拆、帮你存、帮你找，**你只负责最有价值的那一步：思考和创作。**

区别于市面所有 AI 写作工具的核心壁垒：**它了解你的写作方法论，它的输出像你，而非泛泛的模板填充。**

---

## 二、用户真实画像与痛点

| 痛点 | 真实案例 |
|------|---------|
| **读得慢** | 2000+ 字文章认真读需 10 分钟以上，且读完只记得大概 |
| **记不住** | 当时知道讲了什么观点/案例，但具体细节模糊，引用时要从头翻 |
| **找不到** | 微信收藏夹里几十篇文章散落各处，无结构化检索 |
| **虚假掌握感** | 读 AI 摘要觉得"嗯这个我知道"，但实际没读原文，长期损害判断力 |
| **工具割裂** | 整理了一会儿觉得效率太低放弃了，没有合适工具 |
| **写作不像自己** | 市面 AI 写作工具输出的内容没有个人风格，像机器人 |

**用户不是从零开始**：已有一批公众号链接、PDF、旧笔记等待迁移，目标是让它们"活起来"而非"压箱底"。

## 三、核心工作流

```
输入 URL / 上传文档
       │
       ▼
┌─────────────────────┐
│    URL 解析路由器     │
│                     │
│  微信公众号文章？     │──→ Jina Reader API 转 Markdown
│  知乎 / 36kr / 一般网页？│──→ Jina Reader API 转 Markdown（统一兜底）
│  PDF / Word？        │──→ 解析器提取正文
│  付费墙 / 需登录？   │──→ 提示无法抓取（不美化，不造假）
└──────────┬──────────┘
           ▼
      清洗后的纯文本 + 元信息（标题/作者/发布时间/来源平台）
           │
           ▼
┌─────────────────────┐
│    AI 深度解析引擎    │
│                     │
│  1. 原文质量评级      │──→ 直接指出原文论证薄弱/逻辑不通之处（不美化）
│  2. 结构化分析卡片    │──→ 标签/分类/写作方向
│  3. 原始精华摘录      │──→ 原文段落级保留（保留语境，非纯结构化）
│  4. 写作手法学习      │──→ "从这篇你学到了什么写作手法？"
└──────────┬──────────┘
           │
           ▼
   ┌──────┴──────┐
   ▼             ▼
 展示卡片     归档存储
（即时反馈）  （直接写入飞书）
   │             │
   ▼             ▼
 复制 / 归档   飞书多维表格（统一数据中心）
```

## 四、AI 解析输出设计（核心：双重格式）

> **设计原则**：结构化字段便于检索，原始段落便于感受语境。

### 4.1 分析卡片格式

```json
{
  "article_meta": {
    "title": "为什么大部分人的努力是无效的",
    "author": "刘润",
    "source": "公众号",

    "published_at": "2024-12-15",
    "url": "https://mp.weixin.qq.com/s/..."
  },
  "quality_rating": {

    "concerns": ["第三段因果推断略显仓促", "结尾行动建议偏泛"]
  },
  "structured_analysis": {
    "tags": ["#认知", "#效率", "#方法论"],
    "core_arguments": [
```

### 4.2 写作手法学习机制

每次解析完成后，AI 追加提问：

> **"从这篇文章中，你学到了哪些写作手法？结合你自己的写作习惯，你下次会怎么用？"**

用户回答（可选） → 存入"写作方法论库"，写作 Agent 输出时参照，确保原创性而非模仿。

## 五、存储架构（核心：存储层 ≠ 展示层）

```
┌─────────────────────────────────────────────────────┐
│                    存储层（对用户透明）                │
│                                                     │
│  SQLite + ChromaDB（本地向量数据库）                  │
│  ├─ articles 表：文章原文 + 元信息                    │
│  ├─ embeddings 表：文章向量（用于语义检索）            │
│  ├─ essence 表：观点/案例/金句（独立检索单元）         │
│  ├─ writing_methods 表：用户认同的写作方法论           │
│  └─写作素材关联表：素材 ↔ 文章 ↔ 写作方向的多对多关系   │
└─────────────────────────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────┐
│                   展示层（飞书多维表格）               │
│                                                     │
│  文章库：每篇解析后自动写入，支持按来源/标签筛选        │
│  观点库：从所有文章中提取的正反观点，支持对比            │
│  案例库：按行业/场景分类，写文章时主题检索              │
│  金句库：按情感/场景分类，标注出处，避免抄袭风险        │
└─────────────────────────────────────────────────────┘
```

**为什么不是纯飞书？**
- 飞书表格是"展示层"，不是"检索层"。表格无法高效处理"一篇文章 → 多个观点 → 多个金句"的一对多关系，也无法做向量语义检索。
- ChromaDB 是本地轻量向量库（~30MB），无需额外部署服务，随项目交付。

## 六、批量迁移机制

针对用户"有大量存货"的现实情况：

1. **批量提交**：一次粘贴多个 URL（支持换行分隔）
2. **后台队列**：FastAPI 任务队列，后台逐个解析，不阻塞 UI
3. **进度通知**：解析完成时推送通知（WebSocket 或轮询）
4. **元数据复用**：历史文章允许手动补全"发布时间/作者"等元信息

## 七、灵感检索与写作辅助

### 7.1 语义检索流程

```
用户输入写作主题："延迟满足"
        │
        ▼
  ChromaDB 语义搜索
  ├─ 相关观点（5条）：来自哪篇文章，核心论点是什么
  ├─ 相关案例（3个）：人物/数据/转折点
  ├─ 相关金句（5条）：原文摘录，标注出处
  └─相关文章链接：用户可跳转查看原文
        │
        ▼
   打包呈现 + 写作方向建议
        │
        ▼
   用户选择素材 → AI 生成文章大纲（可选）
```

### 7.2 写作分身工作流

**不是直接模仿你的文风**，而是基于"写作方法论库"确保输出在你的审美框架内：

```
用户选择素材 + 写作主题
        │
        ▼
  基于"写作方法论库"生成大纲
  （参照你认同的"先破后立"等结构手法）
        │
        ▼
  用户确认 / 调整大纲
        │
        ▼
  逐段写作，每段完成后标注"使用了XX写作手法"
        │
        ▼
  整体润色，输出符合用户审美框架的初稿
```

## 八、技术栈

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| **前端** | HTML + Vanilla JS | 极简静态页面，无需构建工具，快速交付 |
| **后端** | Python FastAPI | 轻量异步框架，支持 SSE 流式输出 |
| **内容抓取** | Jina Reader API (`r.jina.ai/URL`) | 统一将任意网页转 Markdown，公众号/知乎/付费墙全覆盖 |
| **本地向量库** | SQLite + ChromaDB | 轻量，无需额外部署 |
| **LLM 解析** | Claude / GPT-4 | 结构化分析 + 原文质量评级 |
| **归档存储** | 飞书多维表格 API | 用户可见层，自动同步 |

## 九、实施路线

### Phase 1：MVP（方案 B 第一阶段）
- [ ] 单次 URL / 文件解析 → 分析卡片展示
- [ ] 分析卡片支持一键复制 Markdown
- [ ] SQLite 本地存储（文章 + 解析结果）
- [ ] AI 质量评级 + 写作手法学习提问

### Phase 2：批量 + 归档
- [ ] 批量 URL 提交 + 后台队列解析
- [ ] 飞书多维表格 API 集成（文章库 / 观点库 / 案例库 / 金句库）
- [ ] ChromaDB 向量库集成 + 语义检索

### Phase 3：写作分身
- [ ] 写作方法论库建立（用户回答累积）
- [ ] 灵感检索 UI（主题 → 素材打包）
- [ ] 写作 Agent：选题建议 → 大纲生成 → 逐段写作 → 整体润色

## 十、验伪清单（上线前必答）

- [ ] AI 解析出来的"原始精华摘录"，读起来有没有"灵魂"，是否像人写的而非机器摘要？
- [ ] 批量迁移 10 篇文章后，灵感检索的结果是否相关且可用？
- [ ] 飞书多维表格中的数据结构是否符合你的使用习惯？
- [ ] "写作分身"输出的初稿，是否能达到你"署名发布"的质量标准？
- [ ] 工具本身有没有打断你的阅读心流？（归档是否在 3 秒内完成？）

---

> **最后一句话**：OneEditor 的目标不是帮你读更多文章，而是让你对每篇值得拆解的文章，都拆得更深、用得更活、写得更像自己。
