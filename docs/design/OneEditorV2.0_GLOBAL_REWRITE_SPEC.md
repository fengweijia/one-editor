# OneEditor v2.0 — 产品设计文档

---

## 一、核心价值主张

**把"读"和"拆"文章的时间压缩 80%，构建一个可检索的个性化灵感写作系统。**

本质不是一个"AI 摘要工具"，而是一个 **"阅读代理"** —— 它帮你读、帮你拆、帮你存、帮你找，**你只负责最有价值的那一步：思考和创作。**

区别于市面所有 AI 写作工具的核心壁垒：**它了解你的写作方法论，它的输出像你，而非泛泛的模板填充。**

---

## 二、用户真实画像与痛点

| 痛点 | 真实案例 |
|------|---------|
| **读得慢** | 2000+ 字文章认真读需 10 分钟以上，且读完只记得大概 |
| **记不住** | 当时知道讲了什么观点/案例，但具体细节模糊，引用时要从头翻 |
| **找不到** | 微信收藏夹里几十篇文章散落各处，无结构化检索 |
| **虚假掌握感** | 读 AI 摘要觉得"嗯这个我知道"，但实际没读原文，长期损害判断力 |
| **工具割裂** | 整理了一会儿觉得效率太低放弃了，没有合适工具 |
| **写作不像自己** | 市面 AI 写作工具输出的内容没有个人风格，像机器人 |

**用户不是从零开始**：已有一批公众号链接、PDF、旧笔记等待迁移，目标是让它们"活起来"而非"压箱底"。

## 三、核心工作流

```
输入 URL / 上传文档
       │
       ▼
┌─────────────────────┐
│    URL 解析路由器     │
│                     │
│  微信公众号文章？     │──→ Jina Reader API 转 Markdown
│  知乎 / 36kr / 一般网页？│──→ Jina Reader API 转 Markdown（统一兜底）
│  PDF / Word？        │──→ 解析器提取正文
│  付费墙 / 需登录？   │──→ 提示无法抓取（不美化，不造假）
└──────────┬──────────┘
           ▼
      清洗后的纯文本 + 元信息（标题/作者/发布时间/来源平台）
           │
           ▼
┌─────────────────────┐
│    AI 深度解析引擎    │
│                     │
│  1. 原文质量评级      │──→ 直接指出原文论证薄弱/逻辑不通之处（不美化）
│  2. 结构化分析卡片    │──→ 标签/分类/写作方向
│  3. 原始精华摘录      │──→ 原文段落级保留（保留语境，非纯结构化）
│  4. 写作手法学习      │──→ "从这篇你学到了什么写作手法？"
└──────────┬──────────┘
           │
           ▼
   ┌──────┴──────┐
   ▼             ▼
 展示卡片     归档存储
（即时反馈）  （直接写入飞书）
   │             │
   ▼             ▼
 复制 / 归档   飞书多维表格（统一数据中心）
```

## 四、AI 解析输出设计（核心：双重格式）

> **设计原则**：结构化字段便于检索，原始段落便于感受语境。

### 4.2 写作手法学习机制

每次解析完成后，AI 追加提问：

> **"从这篇文章中，你学到了哪些写作手法？结合你自己的写作习惯，你下次会怎么用？"**

用户回答（可选） → 存入"写作方法论库"，写作 Agent 输出时参照，确保原创性而非模仿。

## 五、存储架构（核心：飞书即 Database）

```
┌─────────────────────────────────────────────────────┐
│                 飞书多维表格（Bitable）                 │
│                 (统一的存储与展示中心)                   │
│                                                     │
│  ├─ 文章库（Articles）：文章原文链接 + 元信息 + 质量评级 │
│  ├─ 观点库（Arguments）：核心论点 + 论据 + 关联文章       │
│  ├─ 案例库（Cases）：案例总结 + 关键细节 + 适用角度        │
│  ├─ 金句库（Quotes）：原文摘录 + 上下文 + 关联文章        │
│  └─ 写作手法库（Techniques）：用户认同的写作方法论         │
└─────────────────────────────────────────────────────┘
```

**为什么砍掉本地数据库（SQLite/ChromaDB）？**
- 飞书多维表格的 API 已经足够强大，完全可以作为中小型项目的无服务器后端（Serverless DB）。
- 对于个人写作者，数据量通常在千级别以内，飞书原生的搜索和过滤功能完全够用，不需要杀鸡用牛刀引入复杂的向量数据库。
- 保证了数据的唯一真实来源（Single Source of Truth），避免了本地与云端数据同步的脑裂问题。
- 飞书多维表格原生的画廊视图、看板视图能提供极佳的"灵感卡片"视觉体验（Aha Moment）。

## 六、批量迁移机制

针对用户"有大量存货"的现实情况：

1. **批量提交**：一次粘贴多个 URL（支持换行分隔）
2. **后台队列**：FastAPI 任务队列，后台逐个解析，不阻塞 UI
3. **进度通知**：解析完成时推送通知（WebSocket 或轮询）
4. **元数据复用**：历史文章允许手动补全"发布时间/作者"等元信息

## 七、灵感检索与写作辅助

### 7.1 灵感检索与组装（非全自动生成）

```
用户输入写作主题："延迟满足"
        │
        ▼
  利用飞书原生搜索 / 后端 API 过滤
  ├─ 相关观点（5条）：来自哪篇文章，核心论点是什么
  ├─ 相关案例（3个）：人物/数据/转折点
  └─ 相关金句（5条）：原文摘录，标注出处
        │
        ▼
   打包呈现 + 写作方向建议
        │
        ▼
   用户像"拼图"一样自己挑选素材，进行组装创作
```

### 7.2 写作手法隐性学习

**不依赖用户手动回答**，而是通过用户的实际使用动作：

```
用户在写《论坚持》时，调用了文章A的"先破后立"结构
        │
        ▼
系统自动记录：该用户偏好"先破后立"的叙事结构
        │
        ▼
不断丰富飞书的"写作手法库"
```

## 八、技术栈

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| **前端** | HTML + Vanilla JS + Tailwind | 极简静态页面，快速交付，注重防爹味卡片设计 |
| **后端** | Python FastAPI | 轻量异步框架，支持 SSE 流式输出 |
| **内容抓取** | Jina Reader API (`r.jina.ai/URL`) | 统一将任意网页转 Markdown |
| **数据库** | 飞书多维表格（Bitable） | 唯一的云端数据库，兼作展示层 |
| **LLM 解析** | Claude / GPT-4 | 结构化分析 + 原文质量评级 |

## 九、实施路线

### Phase 1：MVP (重仓解析与展示)
- [ ] 单次 URL / 文件解析 → 极简、防爹味的分析卡片展示
- [ ] 分析卡片支持一键复制 Markdown
- [ ] AI 质量评级 + 原始精华摘录（保留语境）
- [ ] 打通飞书多维表格 API（建立 5 张关联表）
- [ ] 单篇文章解析后自动拆解写入飞书各表

### Phase 2：批量与资产化
- [ ] 批量 URL 提交 + 后台队列解析
- [ ] 优化飞书多维表格的画廊/看板视图，提升视觉满足感

### Phase 3：灵感唤醒 (暂缓全自动 Agent)
- [ ] 灵感检索 UI：输入主题，从飞书 API 聚合相关素材
- [ ] 隐性记录用户素材调用习惯，丰富写作手法库

## 十、验伪清单（上线前必答）

- [ ] AI 解析出来的"原始精华摘录"，读起来有没有"灵魂"，是否像人写的而非机器摘要？
- [ ] 批量迁移 10 篇文章后，灵感检索的结果是否相关且可用？
- [ ] 飞书多维表格中的数据结构是否符合你的使用习惯？
- [ ] "写作分身"输出的初稿，是否能达到你"署名发布"的质量标准？
- [ ] 工具本身有没有打断你的阅读心流？（归档是否在 3 秒内完成？）

---

> **最后一句话**：OneEditor 的目标不是帮你读更多文章，而是让你对每篇值得拆解的文章，都拆得更深、用得更活、写得更像自己。
